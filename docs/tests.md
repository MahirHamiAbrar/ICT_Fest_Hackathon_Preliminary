# Test Suite Overview

This page indexes the pytest suite under `tests/`. Every test is derived from
a declarative sentence in `README.md` ("Business rules" 1-16 and the "API
contract" section) or the equivalent problem statement
(`docs/problem_statement.md`, Sections 3-5). The mapping is deliberate: each
test file corresponds to one or more business rules / contract clauses, and
each test function encodes exactly one claim from those rules as a black-box
assertion against the running API (via `fastapi.testclient.TestClient`,
same technique as the pre-existing `tests/test_smoke.py`).

Per-file detail pages live in `docs/tests/*.md` and describe, for each test
function: what it asserts, and — where the test currently fails — which real
bug in `app/` it is catching (cross-referenced to the corresponding
`docs/app/*.md` page, which documents the same bug from the implementation
side).

## How to run

```bash
pip install -r requirements.txt
pytest tests/ -q
```

The suite talks to the app in-process (no server needs to be running). Some
tests spin up real thread pools to exercise the "holds under concurrent
requests" clauses in rules 3-7 and 14, and a few (rate limiting) intentionally
send bursts of requests, so a full run takes a few minutes rather than
seconds — this is expected.

`tests/conftest.py` also switches the SQLite connection to WAL journal mode
with a generous busy timeout. This is a test-harness-only setting (not an
application change): hammering a single SQLite file with dozens of
concurrent threads under the default rollback-journal mode produces spurious
`"attempt to write a readonly database"` errors that are an artifact of the
test load, not a business-rule violation. With WAL enabled, the failing set
below is fully deterministic across repeated runs.
