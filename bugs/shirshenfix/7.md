## Bug 7 — Reference Code Uniqueness Race

**Difficulty:** Easy  
**File:** `app/services/reference.py`

### Symptom

Rule 7 requires every booking `reference_code` to be unique, including under
concurrent creation.  
Under concurrency, multiple bookings could receive the same reference code.

Bug #2: "Every booking's reference code is unique, including under concurrent creation." This column has index=True but no unique=True. Without a DB-level unique constraint, nothing stops two concurrent booking creations from persisting the same reference code — whatever generation logic lives in services/ (probably reference_codes.py) can only prevent collisions if it does something like generate-then-insert-and-catch-IntegrityError-and-retry, and that pattern only works if the constraint actually exists at the DB layer. Right now a race would just silently succeed with a duplicate. 


### files and explanations

`next_reference_code` performed a read-modify-write on `_counter["value"]`
without synchronization.  
With concurrent calls, multiple threads could read the same value before any
thread wrote back the incremented value.

### Bug

```python
# Wrong — unsynchronized counter read-modify-write
current = _counter["value"]
_format_pause()
_counter["value"] = current + 1
return f"CW-{current:06d}"
```

Bug 2:
```python
reference_code = Column(String, nullable=False, index=True)
```

### Fix

```python
# Correct — serialize issuance with a lock
with _counter_lock:
    current = _counter["value"]
    _format_pause()
    _counter["value"] = current + 1
    return f"CW-{current:06d}"
```

Bug 2:
```python
reference_code = Column(String, unique=True, nullable=False, index=True)
```
